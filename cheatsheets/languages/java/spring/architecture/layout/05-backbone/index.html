Project Layout Cheatsheet (edges ‚Üî core)¶

A compact, repeatable map for multi-entity Spring Boot apps. Use vertical slices (features) with the same rhythm in each slice: web ‚Üí app ‚Üí domain, plus infra and resources.

Top-level¶

<app>/
‚îú‚îÄ build.gradle | pom.xml
‚îú‚îÄ README.md
‚îú‚îÄ .editorconfig
‚îú‚îÄ .gitignore
‚îî‚îÄ src/

Keep the repo boring: one app per repo until you truly need multiple deployables.

Package skeleton (repeat per slice)¶

src/main/java/com/edge/<app>/
‚îú‚îÄ web/                # HTTP edge: controllers, DTOs, ProblemDetail
‚îÇ  ‚îî‚îÄ {slice}/
‚îÇ     ‚îú‚îÄ {Slice}Controller.java
‚îÇ     ‚îú‚îÄ dto/
‚îÇ     ‚îÇ  ‚îú‚îÄ Create{Thing}Request.java
‚îÇ     ‚îÇ  ‚îú‚îÄ Update{Thing}Request.java
‚îÇ     ‚îÇ  ‚îî‚îÄ {Thing}Response.java
‚îÇ     ‚îî‚îÄ mapper/
‚îÇ        ‚îî‚îÄ {Slice}WebMapper.java          # MapStruct, edge-owned mappings
‚îú‚îÄ app/                # Use-cases (application layer)
‚îÇ  ‚îî‚îÄ {slice}/
‚îÇ     ‚îú‚îÄ Create{Thing}UseCase.java
‚îÇ     ‚îú‚îÄ Update{Thing}UseCase.java
‚îÇ     ‚îú‚îÄ Query{Thing}UseCase.java
‚îÇ     ‚îî‚îÄ {Slice}Service.java               # @Service, @Transactional, implements interfaces
‚îú‚îÄ domain/             # Entities + repository ports (interfaces)
‚îÇ  ‚îú‚îÄ {slice}/
‚îÇ  ‚îÇ  ‚îú‚îÄ {Thing}.java
‚îÇ  ‚îÇ  ‚îî‚îÄ {Thing}Repository.java
‚îÇ  ‚îî‚îÄ common/
‚îÇ     ‚îú‚îÄ BaseEntity.java
‚îÇ     ‚îú‚îÄ Money.java / Quantity.java        # value objects (optional)
‚îÇ     ‚îî‚îÄ DomainEvent.java (optional)

Slices are feature folders like catalog/, inventory/, sales/, procurement/. Add more as the domain grows.

Resources & migrations¶

src/main/resources/
‚îú‚îÄ application.yml
‚îú‚îÄ application-<profile>.yml                # local, test, prod
‚îú‚îÄ db/migration/
‚îÇ  ‚îú‚îÄ V1__init_<slice>.sql
‚îÇ  ‚îú‚îÄ V2__add_<constraint>.sql
‚îÇ  ‚îî‚îÄ ...
‚îî‚îÄ logback-spring.xml

Truth lives in Flyway. Entities follow the SQL, not vice-versa.

Error handling (edge)¶

web/error/
‚îú‚îÄ GlobalExceptionHandler.java              # @RestControllerAdvice ‚Üí ProblemDetail
‚îî‚îÄ ProblemTypes.java                        # stable type URIs (use properties for base URL)
  • Translate exceptions once at the web edge.
  • Use stable type URIs like /problems/{slice}/{problem}; base set via properties.

Tests mirror main¶

src/test/java/com/edge/<app>/
‚îú‚îÄ web/{slice}/{Slice}ControllerIT.java     # @SpringBootTest + MockMvc
‚îú‚îÄ app/{slice}/{Slice}ServiceTest.java      # unit tests (Mockito) or @DataJpaTest if needed
‚îî‚îÄ domain/{slice}/{Thing}RepositoryIT.java  # @DataJpaTest
src/test/resources/application-test.yml     # H2 or Testcontainers

Responsibilities & allowed directions¶

  • web/ Speaks HTTP and DTOs. Handles validation, ProblemDetail, headers (Location, caching). May depend on: app, domain (types only), infra.mapping (MapStruct config). Never: return entities, call repositories directly.

  • app/ Orchestrates use-cases, transactions, cross-entity rules. Throws domain/app exceptions. May depend on: domain (entities, repositories). Never: know about HTTP, controllers, servlet stuff.

  • domain/ Entities, value objects, repository ports (Spring Data interfaces are fine). Self-contained invariants. No Spring web annotations. Never: know about web or infra.

  • infra/ Adapters (DB config, messaging, security), technical mappings/projections. May depend on: domain for entity classes. Never: call controllers or use DTOs.

Think downward-only: web ‚Üí app ‚Üí domain (and infra supports the stack).

Naming rules (keep it predictable)¶

  • DTOs: {Action}{Thing}Request (input), {Thing}Response (output). Scenario-first names: AdjustStockRequest, not UpdateWarehouse.
  • Use-cases: verbs: CreateProductUseCase, ListProductsUseCase. Services implement multiple use-cases per slice: CatalogService implements CreateProductUseCase, ...
  • Repositories: {Thing}Repository. No ‚ÄúDAO‚Äù suffixes.
  • Controllers: {Thing}Controller or {Slice}Controller if it aggregates.

Mappers (at the edge)¶

  • Web mappers live under web/{slice}/mapper. They convert DTO ‚Üî Command and Entity ‚Üî Response.
  • Global defaults in infra/mapping/MapStructConfig.java:
@MapperConfig(
  componentModel = "spring",
  unmappedTargetPolicy = ReportingPolicy.ERROR,
  nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE
)

Annotate each mapper with @Mapper(config = MapStructConfig.class).

Properties & configuration¶

  • Put base URIs, feature flags, cache TTLs in application.yml.
  • Bind via @ConfigurationProperties(prefix = "app") classes under infra/ or web/config/ (if HTTP-specific).
  • Profiles: local, test, prod. Never commit secrets to YAML.

Where future files go (decision table)¶

Thing you‚Äôre adding Folder Notes

Input/Output models for that endpoint web/{slice}/dto/ Validate with Bean Validation
Mapping between DTOs and entities web/{slice}/mapper/ Use MapStruct
New business workflow app/{slice}/ Add *UseCase + implement in service
Cross-entity rule (e.g., stock never negative) app/{slice}/{Slice}Service.java Enforce in a @Transactional use-case
New entity domain/{slice}/{Thing}.java Keep invariants here

Projections/Specifications infra/persistence/ Don‚Äôt leak to web/app

Scheduled job / batch app/{slice}/ or infra/ (adapter) Use-case in app, triggers in infra
Integration client (SMTP, S3, Kafka) infra/{integration}/ One subpackage per integration

Adding a new slice (recipe)¶

  1. Create domain/{slice} with entities + repositories.
  2. Create app/{slice} with use-case interfaces + {Slice}Service.
  3. Create web/{slice} with controller, DTOs, mapper.
  4. Add Flyway migration(s).
  5. Add tests mirroring main.

You can ship only web+app+domain for a slice; infra stays minimal until needed.

When to split things further¶

  • A service file passes ~400‚Äì500 lines or mixes unrelated workflows ‚Üí split by verb or read/write (OrderCommandService vs OrderQueryService).
  • DTO package gets crowded ‚Üí subfolders per aggregate (dto/product/*, dto/category/*).
  • Query logic turns gnarly ‚Üí move complex queries to Specifications/Query classes in infra/persistence.

Guardrails (the ‚Äúdon‚Äôt get weird‚Äù list)¶

  • Don‚Äôt return entities from controllers. Always map to responses.
  • Don‚Äôt inject repositories into controllers. Controllers talk to use-cases only.
  • Don‚Äôt let web know about JPA annotations; don‚Äôt let domain know about HTTP.
  • Don‚Äôt put shared junk in domain/common/. Only true, reusable domain primitives belong there.

Tiny examples (paths)¶

  • Create product endpoint web/catalog/ProductController#create DTOs in web/catalog/dto/CreateProductRequest.java Mapper in web/catalog/mapper/ProductWebMapper.java Use-case app/catalog/CreateProductUseCase.java ‚Üí implemented by CatalogService.

  • Adjust stock web/inventory/StockItemController#adjust AdjustStockRequest ‚Üí InventoryService.adjustStock(...) ‚Üí StockRepository.
