üß† Python Virtual Environments (venv) ‚Äî Cheatsheet¶

üß© What It Is¶

venv means virtual environment. It‚Äôs a self-contained Python environment that lives inside your project folder. Think of it as a sandbox where you can install Python packages without messing up your system or other projects.

Each venv:

  • Has its own Python interpreter (copied from your system‚Äôs).
  • Has its own site-packages folder (where libraries get installed).
  • Ignores all global packages unless told otherwise.

üéØ Why Use It¶

Without venv, if you install libraries like this:

pip install mkdocs

they go into your global Python ‚Äî polluting everything. Soon you‚Äôll have version conflicts and dependency chaos between projects.

With venv:

  • Every project has isolated dependencies.
  • You can safely use different library versions per project.
  • You can delete or recreate it anytime ‚Äî your system stays clean.

‚öôÔ∏è How To Create and Use¶

1. Create a venv

python3 -m venv .venv

This makes a folder .venv with a fresh Python setup inside.

2. Activate it

System Command
Linux / macOS source .venv/bin/activate
Windows (cmd) .venv\Scripts\activate
Windows (PowerShell) .venv\Scripts\Activate.ps1

When active, your shell prompt changes ‚Äî usually you‚Äôll see (.venv) in front.

3. Install libraries

pip install mkdocs mkdocs-material mkdocs-obsidian

They‚Äôll install inside .venv only, not globally.

4. Freeze (optional) If you want to share exact versions with others:

pip freeze > requirements.txt

Then anyone can recreate your setup with:

pip install -r requirements.txt

5. Deactivate

deactivate

üßπ In .gitignore¶

You always exclude .venv/:

.venv/

Because it‚Äôs bulky, system-specific, and can be recreated from requirements.txt.

üß† Quick Summary Table¶

Task Command Notes
Create python3 -m venv .venv Makes environment in .venv/
Activate source .venv/bin/activate Start using it
Deactivate deactivate Return to system Python
Install packages pip install <package> Only affects this project
List packages pip list Shows installed inside venv
Export dependencies pip freeze > requirements.txt For version tracking
Reinstall from file pip install -r requirements.txt Recreates setup

üß© Mental Model¶

Imagine your computer is a city. Each Python project is its own apartment. venv gives each apartment:

  • Its own kitchen (libraries)
  • Its own utilities (interpreter)
  • But shares the city‚Äôs infrastructure (OS-level Python)

So if one apartment burns its kitchen down (dependency conflict), the others stay safe.

‚ö†Ô∏è venv Troubleshooting & Common Pitfalls¶

üß© 1. ‚ÄúI installed a package, but Python says ‚ÄòModuleNotFoundError‚Äô‚Äù¶

Cause: You forgot to activate your virtual environment. Fix: Activate it before installing or running:

source .venv/bin/activate

You‚Äôll know it‚Äôs active when you see (.venv) before your prompt.

üêç 2. ‚ÄúI have multiple Python versions ‚Äî which one is used?‚Äù¶

Each venv is tied to the Python version that created it. Example:

python3.11 -m venv .venv

means your venv uses Python 3.11, even if your system later upgrades to 3.12. If you remove or change your system Python, that venv might break.

Tip: Always note your Python version in your README.

üíÄ 3. ‚ÄúThe venv folder got corrupted or too big‚Äù¶

Don‚Äôt panic ‚Äî just delete .venv/ and recreate it:

rm -rf .venv
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

That‚Äôs it. Your clean sandbox returns, good as new.

üß† 4. ‚ÄúMkDocs (or some tool) not found even though I installed it‚Äù¶

That means the tool was installed inside the venv, but your shell isn‚Äôt using that path. Fix: activate the venv before running:

source .venv/bin/activate
mkdocs serve

or run it explicitly from inside:

.venv/bin/mkdocs serve

üíº 5. ‚ÄúDo I need to commit .venv to Git?‚Äù¶

Never. Just commit your requirements.txt. .venv is like your temporary workbench ‚Äî everyone can rebuild it from the list of dependencies.

‚öôÔ∏è 6. ‚ÄúCan I have more than one venv?‚Äù¶

Absolutely. You can have one per project, or even multiple inside one project for different experiments:

.venv/
.venv-py311/
.venv-py312/

Just activate whichever one you need.

üîÑ 7. ‚ÄúI closed the terminal ‚Äî do I have to activate again?‚Äù¶

Yes. The activation is per shell session. Once you close it, your environment resets, and you‚Äôll need to:

source .venv/bin/activate

again when reopening your project.

üß† Quick Diagnosis Checklist¶

If something isn‚Äôt working:

  1. Is your prompt showing (.venv)? ‚Üí If not, activate it.
  2. Does which python or where python show .venv/bin/python? ‚Üí If not, wrong environment.
  3. Did you reinstall dependencies after deleting .venv? ‚Üí Run pip install -r requirements.txt.

üí° Golden Rule¶

When in doubt:

Delete .venv, recreate it, reinstall. It‚Äôs faster than debugging a broken environment.

‚öôÔ∏è Advanced venv Tricks & Integrations¶

üß† 1. Auto-activate in VS Code¶

VS Code has native support for virtual environments.

  1. Open your project folder in VS Code (the one with .venv/).
  2. Open Command Palette ‚Üí Python: Select Interpreter.
  3. Choose the one that points to .venv/bin/python (or .venv\Scripts\python.exe on Windows).

Now every terminal inside VS Code auto-activates the venv, and ‚ÄúRun‚Äù or ‚ÄúDebug‚Äù commands use it automatically.

Tip: Add this to your project‚Äôs .vscode/settings.json (optional, but nice):

{
  "python.defaultInterpreterPath": ".venv/bin/python"
}

üêö 2. Auto-activate when entering folder (shell-level trick)¶

If you live in your terminal, you can make the venv activate automatically whenever you cd into the project.

For bash or zsh:¶

Append to your ~/.bashrc or ~/.zshrc:

# Auto-activate venv when entering project folder
cd() {
  builtin cd "$@" || return
  if [ -f ".venv/bin/activate" ]; then
    source .venv/bin/activate
  fi
}

That makes the environment self-starting ‚Äî no more ‚Äúforgot to activate‚Äù moments.

üê≥ 3. Use venv in Docker (for consistent builds)¶

You usually don‚Äôt need venv inside Docker because Docker itself isolates environments ‚Äî but it‚Äôs sometimes useful when developing locally and deploying via container.

A typical lightweight pattern:

FROM python:3.11-slim

WORKDIR /app
COPY . .
RUN python3 -m venv /venv
ENV PATH="/venv/bin:$PATH"

RUN pip install --upgrade pip && pip install -r requirements.txt
CMD ["mkdocs", "serve", "-a", "0.0.0.0:8000"]

This way, the container runs everything inside /venv, giving full reproducibility between your laptop and the container.

üîÑ 4. Use venv in CI/CD (GitHub Actions, GitLab, etc.)¶

Virtual environments integrate beautifully in automation. Example GitHub Actions job for MkDocs:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: python -m venv .venv
      - run: source .venv/bin/activate && pip install -r requirements.txt
      - run: source .venv/bin/activate && mkdocs build

That ensures each run is clean, reproducible, and isolated ‚Äî no dependency drift over time.

üß© 5. Use venv executables directly (no activation needed)¶

Sometimes automation scripts shouldn‚Äôt depend on interactive activation.

You can run any installed tool directly from inside .venv/bin:

.venv/bin/mkdocs serve

That works even if the venv isn‚Äôt ‚Äúactivated.‚Äù It‚Äôs cleaner in scripts, cron jobs, or Dockerfiles.

üßπ 6. Clean up and reset your environment fast¶

When you want a fresh start:

rm -rf .venv site/ __pycache__/
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

That gives you a sterile lab again, like hitting ‚Äúreset‚Äù on a scientific experiment.

üß† Rule of Thumb for Power Users¶

  • Local dev ‚Üí always use .venv
  • Dockerized app ‚Üí system Python inside container is fine, or one global /venv
  • CI/CD ‚Üí recreate venv per run for reliability

Next level from here would be environment management tools like:

  • pip-tools ‚Üí for precise dependency pinning
  • poetry ‚Üí combines venv + dependency management + packaging
  • pyenv ‚Üí handles multiple Python versions on one system
