üåê Web State Evolution (told through Spring‚Äôs lens)¶

How the web learned to remember ‚Äî and how that memory evolved from cookies to tokens, from servers to clients.

1) Why Web State Exists¶

HTTP was designed to be stateless ‚Äî every request is a blank slate. Without extra mechanisms, the server can‚Äôt tell whether two requests come from the same user.

To make web experiences persistent ‚Äî logins, carts, dashboards ‚Äî we need a state bridge between requests. That bridge has changed over time, but the problem never did.

2) Cookies ‚Äî The Original Memory Hack¶

A cookie is a tiny key‚Äìvalue pair that a server asks the browser to keep and send back with future requests.

Server ‚Üí Set-Cookie: sessionId=abc123
Browser ‚Üí Cookie: sessionId=abc123

Cookies make HTTP feel stateful, but they‚Äôre just labels, not memory. The actual data lives on the server, keyed by that label:

sessionId=abc123 ‚Üí user=Edgaras, cart=3 items

Think of it as a coat-check system:

  • You (the browser) keep the ticket.
  • The server keeps your coat (the session data).

When you return, you hand in the ticket; the server finds your coat.

If either side forgets (cookie cleared or session expired), you start fresh.

3) Why the Client Can‚Äôt Be Trusted¶

Cookies are stored client-side, so they can be read, changed, or deleted. That‚Äôs why cookies must be protected with strict attributes:

Attribute Protects Against Description
HttpOnly JavaScript access JS can‚Äôt read or modify it
Secure Network sniffing Only sent via HTTPS
SameSite CSRF attacks Controls cross-site sending

Servers must never trust cookie data blindly ‚Äî it‚Äôs a reference, not proof.

4) The Stateful Model ‚Äî Server Remembers You¶

Example: Spring MVC¶

Old-school web applications used server-side sessions.

POST /login ‚Üí Set-Cookie: JSESSIONID=xyz123

Spring stores that session internally:

xyz123 ‚Üí userId=42, username="edgaras"

Next request:

GET /profile
Cookie: JSESSIONID=xyz123

Spring restores your identity automatically via HttpSession.

Code sketch:

User user = (User) session.getAttribute("user");

This model is stateful ‚Äî the server remembers each user‚Äôs state in memory or a shared store (like Redis).

5) The Stateless Model ‚Äî Client Proves Itself¶

Modern REST APIs abandoned server memory. Instead of ‚Äúthe server remembers you,‚Äù we now do ‚Äúyou prove who you are each time.‚Äù

The token (e.g., JWT) is:

  • Signed by the server
  • Self-contained (includes user info, expiration, roles)
  • Verifiable without server memory

No lookup tables. No HttpSession. Every request is independent and verifiable.

This makes scaling simple ‚Äî any server can validate the token.

6) Why This Change Happened¶

The shift wasn‚Äôt stylistic ‚Äî it was infrastructural.

Old model (sessions):

  • Works for single servers.
  • Breaks in load-balanced or cloud environments (session stickiness issues).

New model (tokens):

  • Works with multiple instances.
  • Stateless and cache-friendly.
  • Portable across APIs, apps, and devices.

Tokens are ideal for distributed, multi-client systems. Cookies still shine in browser-native, single-origin web apps.

7) Hybrid World ‚Äî Cookies Meet Tokens¶

Many modern systems combine both ideas:

  • Browser clients: store JWT in an HttpOnly cookie for security
  • Mobile / API clients: store JWT manually and send via Authorization header

Same token, different transport.

Cookies are now often just the delivery vehicle for tokens.

8) Cross-Language Equivalents¶

üß© Java / Spring¶

  • Old: HttpSession + JSESSIONID

Each ecosystem faces the same choice:

10) Core Insight¶

Cookies gave the web its first memory, but tokens gave it freedom ‚Äî freedom to scale, to move between servers, to talk to any device.

Spring just reflects that same journey:

  • From JSESSIONID ‚Üí to JWT ‚Üí to OAuth2/OpenID Connect.

The evolution wasn‚Äôt just technical ‚Äî it was philosophical: moving from the server knows you to you can prove who you are.

11) Practical Developer Takeaways¶

  • Understand cookies and sessions conceptually ‚Äî they‚Äôre still vital for browsers.
  • Build your APIs statelessly ‚Äî each request should carry its own proof (token).
  • Use Spring Security + JWT/OAuth2 as your modern baseline.
  • Learn cross-language equivalents ‚Äî they all rhyme with Spring‚Äôs concepts.
  • For scale: state belongs at the edge (client), not in memory on the server.

12) In a Sentence¶
