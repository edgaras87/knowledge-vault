1. The Old Days ‚Äî Manual Wiring¶

In plain Java, you were the conductor of a one-person orchestra.

UserService user = new UserService();
PaymentService pay = new PaymentService(user);
OrderService order = new OrderService(pay);

You wrote new everywhere. You knew every connection by hand.

That works until your system grows and suddenly:

  • Changing one class breaks five others.
  • You can‚Äôt test one piece without booting everything.
  • You have no central place that knows what depends on what.

Developers started dreaming of a container ‚Äî something that could build and connect objects automatically.

That dream became Spring.

2. Reflection ‚Äî Letting Code Look at Code¶

Java has a secret mirror called reflection. It lets a program peek into its own classes while running:

  • ‚ÄúWhat fields does this class have?‚Äù
  • ‚ÄúDoes it have an annotation called @Component?‚Äù
  • ‚ÄúCan I call this method without knowing its name at compile time?‚Äù

Spring uses that mirror to discover and build your objects.

When you run your app, it quietly scans every class on your classpath. When it finds one marked like this:

It says: Ah, that‚Äôs mine to manage. Then it creates an instance reflectively ‚Äî without you writing new PaymentService().

That‚Äôs where the magic begins: the framework starts taking responsibility for creation.

3. Beans ‚Äî Living Java Objects¶

Spring calls these managed objects beans. They‚Äôre still just Java objects, but with a difference ‚Äî Spring controls their entire life.

A bean is born when the container starts. It‚Äôs injected with whatever it needs. It lives inside a structure called the ApplicationContext, and when the application shuts down, Spring cleans it up.

You can think of the ApplicationContext as a living ecosystem of beans:

Each one can depend on the others ‚Äî but they never create each other directly. They trust Spring to connect them.

4. Dependency Injection ‚Äî Letting Someone Else Plug the Cables¶

In normal Java, a class builds its own tools. With Spring, it simply says what it needs, and the container provides it. That pattern is called Dependency Injection (DI).

  • With Spring: there‚Äôs a tool manager handing out hammers as needed.

You don‚Äôt decide which AuditService to use. Spring does. At startup, it looks at your constructor and says: ‚ÄúI have an AuditService bean ‚Äî I‚Äôll plug it in.‚Äù

That‚Äôs constructor injection ‚Äî the cleanest and safest style. Spring can also inject dependencies through setters or directly into fields, but constructor injection makes dependencies explicit, like stating ingredients on a recipe.

5. Inversion of Control ‚Äî The Big Flip¶

Before Spring, you wrote code like this:

You were in charge of when and how things were created.

Spring flips that ‚Äî hence the phrase Inversion of Control (IoC). Now, you declare what your classes need, and the framework decides when to create, connect, and destroy them.

You stopped being the factory. You became the designer ‚Äî telling the system what components exist and how they should relate. Spring became the conductor of your orchestra.

6. The ApplicationContext ‚Äî Spring‚Äôs Brain¶

The ApplicationContext is Spring‚Äôs central nervous system. When you run a Spring Boot app with:

you‚Äôre really saying: ‚ÄúHey Spring, build my world.‚Äù

Here‚Äôs roughly what happens:

  1. Spring creates the ApplicationContext.
  2. It scans your classpath for components (@Component, @Service, @Repository‚Ä¶).
  3. It builds a list of BeanDefinitions ‚Äî recipes describing how to create each bean.
  4. It uses reflection to instantiate them.
  5. It injects dependencies.
  6. It calls initialization hooks (@PostConstruct).
  7. Finally, it announces: ‚ÄúContext refreshed ‚Äî everything‚Äôs alive.‚Äù

From this point on, you can request any bean, and Spring will hand you a fully wired instance.

7. Bean Lifecycle ‚Äî The Circle of (Application) Life¶

Every bean in Spring goes through predictable stages:

1. Definition  ‚Üí Spring learns about the class
2. Instantiation  ‚Üí Object is created reflectively
3. Dependency Injection  ‚Üí Fields/constructors are filled
4. Initialization  ‚Üí @PostConstruct runs
5. Ready State  ‚Üí Bean is active and serving
6. Destruction  ‚Üí @PreDestroy runs at shutdown

This is why your app can start, serve, and shut down gracefully. Spring takes care of object lifespans so you can focus on behavior.

8. Why This Architecture Matters¶

This container-based system unlocks a few superpowers:

  • Loose coupling ‚Äî classes don‚Äôt depend on each other directly.
  • Configurable behavior ‚Äî you can change how things work via annotations or properties.
  • Automatic lifecycle management ‚Äî no manual setup or cleanup.
  • Testability ‚Äî you can inject mocks for testing instead of real beans.

Once you experience it, going back to plain new feels like riding a horse after owning a spaceship.

9. AOP ‚Äî Teaching Beans New Tricks¶

When Spring controls your objects, it can extend their behavior. That‚Äôs where Aspect-Oriented Programming (AOP) comes in.

Think of AOP as a transparent wrapper around your beans. When someone calls a method, Spring can sneak in extra logic before or after it ‚Äî like logging, transactions, or caching ‚Äî without changing the original class.

    @Transactional

When you call pay(), you‚Äôre actually calling a proxy, not the raw object. That proxy:

  1. Opens a transaction,
  2. Calls your method,
  3. Commits or rolls back depending on the result.

Same story for @Async, @Cacheable, or @Secured ‚Äî all powered by this wrapping trick. That‚Äôs how crosscutting concerns stay separate from business logic.

10. How Spring Pulls It Off¶

Behind the scenes:

  • Reflection lets Spring look at classes and create them dynamically.
  • IoC gives Spring control over who creates what.
  • AOP gives Spring control over how methods behave when called.

Together, they form a vertical architecture:

Reflection ‚Üí builds beans
     ‚Üì
IoC ‚Üí manages relationships and lifecycles
     ‚Üì
AOP ‚Üí adds behavior at runtime

Each layer adds a dimension: first existence, then orchestration, then influence.

11. The Philosophy in One Line¶

Spring‚Äôs goal is to let developers think about ‚Äúwhat should happen,‚Äù not ‚Äúhow to glue it all together.‚Äù

It doesn‚Äôt replace Java ‚Äî it elevates it, turning plain classes into cooperative citizens inside a managed world.

12. The Lifecycle of a Spring Application¶

  1. You press Run.
  2. Spring starts and scans your classes.
  3. It builds a map of all components.
  4. It reflectively creates and connects them.
  5. It wraps some in proxies for AOP.
  6. It publishes ‚ÄúContext Refreshed‚Äù ‚Äî your app is alive.
  7. When you stop, it calls cleanup hooks and shuts down gracefully.

That‚Äôs the entire story ‚Äî an ecosystem built from reflection, orchestration, and trust.

Spring isn‚Äôt a library you call ‚Äî it‚Äôs an environment you live in. It doesn‚Äôt just execute code; it hosts it. Once you understand that, every @Autowired, every @Transactional, every line of your app fits into one graceful rhythm: discover, build, wire, run, extend, and clean up.
