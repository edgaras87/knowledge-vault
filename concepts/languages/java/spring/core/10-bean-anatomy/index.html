üß© Spring Bean Anatomy ‚Äî Definition, Scope, Proxy, and Lifecycle Hooks¶

Essence: A Spring Bean is a managed Java object whose creation, wiring, lifecycle, and sometimes even identity are controlled by the Spring container ‚Äî not by your code. It‚Äôs a living object with metadata, behavior, and reflective wrapping.

üå± The Mental Model¶

A bean in Spring is to the container what a cell is to an organism: it has a definition (DNA), a lifecycle (birth ‚Üí death), and a membrane (proxy) that controls how it interacts with the outside world.

üß¨ 1. Bean Definition ‚Äî the DNA¶

Before a bean exists, Spring builds a BeanDefinition ‚Äî metadata describing how to create it.

@Component ‚Üí scanned
@Bean ‚Üí declared
XML ‚Üí defined

Each definition contains:

Field Description
beanClass The Class<?> object of the bean
scope "singleton", "prototype", "request", etc.
dependencies Other beans to inject
initMethod, destroyMethod Lifecycle callbacks
lazyInit Whether to delay creation until needed
primary, qualifiers Conflict resolution hints

The BeanDefinition lives in memory long before any object is created.

‚öôÔ∏è 2. Instantiation ‚Äî the Birth¶

When the context is refreshed, Spring instantiates the bean via reflection:

Object bean = beanClass.getDeclaredConstructor().newInstance();

Then it resolves all declared dependencies (@Autowired, constructor args, etc.).

At this point:

  • The object exists.
  • Dependencies are injected.
  • Lifecycle callbacks haven‚Äôt yet run.

This is the infant bean stage.

üß© 3. Initialization ‚Äî Becoming Alive¶

Spring then applies post-processing and lifecycle interfaces:

Mechanism Trigger Purpose
@PostConstruct Annotation Custom setup logic
InitializingBean.afterPropertiesSet() Interface Legacy init hook
init-method XML config Alternative init
BeanPostProcessor Global interceptors Modify/replace beans after creation

For example, a post-processor might wrap a bean in a proxy to add behavior (@Transactional, @Async, etc.).

After initialization, the bean is fully active ‚Äî ready to be used or injected elsewhere.

ü™û 4. Proxies ‚Äî The Membrane¶

A proxy is a thin reflective wrapper around your bean. It intercepts method calls to add extra logic like transactions, security, or async execution.

Two main types:¶

Type Tool Used For
JDK Dynamic Proxy Java‚Äôs Proxy API Interfaces only
CGLIB Proxy Bytecode subclass Concrete classes

Example:

@Transactional
public class PaymentService { ... }

Spring replaces it at runtime with something like:

PaymentService$$EnhancerBySpringCGLIB$$abc123

When you call a method, the proxy runs code before and after your logic (start transaction ‚Üí invoke target ‚Üí commit/rollback).

üß≠ 5. Scopes ‚Äî The Lifecycle Context¶

A bean‚Äôs scope defines how long it lives and where it‚Äôs shared:

Scope Description
singleton One shared instance per context (default)
prototype New instance every injection
request One per HTTP request (web only)
session One per HTTP session
application One per servlet context
websocket One per WebSocket session

Example:

@Scope("prototype")
@Component
public class TaskProcessor { ... }

@Autowired of a prototype bean creates a fresh instance each time.

üß© 6. Awareness Interfaces ‚Äî Talking to the Container¶

Some beans want to ‚Äúknow‚Äù about their environment. Spring injects this information through special Aware interfaces:

Interface What you get
ApplicationContextAware Access to the context
BeanNameAware The name assigned to your bean
EnvironmentAware Config and property access
ResourceLoaderAware File/resource utilities

These aren‚Äôt common for business logic, but essential for infrastructure beans (framework internals, logging, plugin systems).

üåá 7. Destruction ‚Äî Graceful Death¶

At shutdown or when the context closes:

Mechanism When
@PreDestroy Before bean destruction
DisposableBean.destroy() Legacy cleanup
destroy-method XML config
ContextClosedEvent Published globally

Spring calls these hooks reflectively to release connections, threads, caches, etc.

üß© 8. The Full Lifecycle Map¶

1. Scan classes ‚Üí build BeanDefinition
2. Instantiate ‚Üí inject dependencies
3. Apply BeanPostProcessors
4. Call @PostConstruct / init-method
5. Bean ready for use
6. (Optionally) wrapped in proxy
7. Serve requests or logic
8. Context shutdown ‚Üí call @PreDestroy

‚ö° Putting it Together ‚Äî Reflection‚Äôs Thread¶

Reflection appears at every layer:

  • Definition stage: reads annotations (Class.isAnnotationPresent)
  • Instantiation stage: calls constructors
  • Injection stage: sets fields or methods
  • Initialization stage: invokes annotated methods
  • Proxying stage: intercepts via dynamic subclasses

Yet once the bean is active, reflection disappears ‚Äî normal method calls dominate.

üß≠ TL;DR Summary¶

Aspect Description Driven by Reflection?
Definition Metadata blueprint ‚úÖ
Instantiation Create object ‚úÖ
Injection Wire dependencies ‚úÖ
Post-processing Modify or proxy ‚úÖ
Active state Business logic ‚ùå
Destruction Cleanup hooks ‚úÖ

ü™û Core takeaway¶

A Spring Bean is not just an object ‚Äî it‚Äôs a meta-object, born from metadata, shaped by reflection, guarded by proxies, and governed by lifecycle rules. Once awakened, it becomes indistinguishable from plain Java ‚Äî until the next context refresh.
